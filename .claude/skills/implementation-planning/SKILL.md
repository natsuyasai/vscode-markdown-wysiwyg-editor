# 実装プラン策定（フェーズ2）

フェーズ1で確定した要求を元に、実装プランを設計する。

## 進め方

1. **設計**: 確定要求を満たす実装方針を設計する。既存の関数・ユーティリティ・パターンを優先的に再利用し、新規追加は最小限にする。必要なら `Plan` サブエージェントで設計を検証する。
2. **複数案の比較と確認**: 妥当なアプローチが複数ある場合は、各案のトレードオフを整理し、`AskUserQuestion` で**どの案で対応するかを必ず確認**する。1案に決まるまで実装に進まない。
3. **一時ドキュメント出力**: 確定したプランを `tmp/plans/` 配下にMarkdownで出力する（git管理外。別セッション・別担当へ引き継ぐため）。
4. **確定**: ユーザーにプランを提示し、着手可能か合意を取る。

## 一時ドキュメントの仕様

- 出力先: `tmp/plans/<日付>-<対象名>.md`（例: `tmp/plans/2026-06-17-outline-search.md`）
- `tmp/` はgit管理外（`.gitignore` 済み）
- 別セッションが単独で実装を再開できる粒度で記述する

### 記載項目

- **背景 / 目的**: なぜこの変更を行うのか（フェーズ1の確定要求）
- **対応方針**: 採用するアプローチ（不採用案の理由も簡潔に）
- **変更対象**: 主要な変更ファイルと、再利用する既存関数・パターン（パス付き）
- **作業分割**: コミット単位に分けた作業ステップ（各ステップでTDDを回す前提）
- **テスト方針**: 単体テスト / Storybookカタログ / play function（画面単体・結合）のどれを作るか
- **完了基準と検証方法**: format / test / build を含む確認手順

## 完了条件

- 採用プランが1案に確定し、ユーザーと合意されている
- `tmp/plans/` に引き継ぎ可能な粒度のドキュメントが出力されている

確定したらフェーズ3（`subagent-tdd-implementation`）へ進む。

## 禁止事項

- 複数案があるのにユーザー確認なしで1案を選んで進めること
- プランを一時ドキュメントに残さず実装に進むこと
